# Unity–ROS 2 Jazzy 시스템 아키텍처

## 1. 기준 구조

```text
macOS 또는 Windows 호스트
┌────────────────────────────────────────────┐
│ Unity Editor                               │
│  ├─ 차동구동 물리                          │
│  ├─ LiDAR / Camera / IMU                   │
│  └─ ROS# WebSocket client                  │
└──────────────────┬─────────────────────────┘
                   │ ws://localhost:9090
┌──────────────────▼─────────────────────────┐
│ Ubuntu 24.04 / ROS 2 Jazzy 컨테이너        │
│  ├─ rosbridge_websocket                    │
│  ├─ bridge_probe                           │
│  ├─ robot_state_publisher                  │
│  ├─ SLAM Toolbox                           │
│  ├─ Nav2                                   │
│  └─ RViz2                                  │
└────────────────────────────────────────────┘
```

## 2. 역할 분리

Unity는 가상 세계, 물리, 렌더링, 센서 원시 데이터를 담당한다. ROS 2는 로봇 상태 추정, 지도 작성, 경로 계획, 행동 제어를 담당한다. RViz2는 ROS 데이터의 해석 화면이지 물리 시뮬레이터가 아니다.

이 구분은 기존 Gazebo 실습과 같다. Unity는 Gazebo의 모든 기능을 자동으로 대체하지 않는다. 필요한 센서와 동역학을 구현해 같은 ROS 인터페이스를 제공해야 한다.

## 3. 최소 토픽 계약

| 토픽 | 방향 | 메시지 | 역할 |
|---|---|---|---|
| `/cmd_vel` | ROS → Unity | `geometry_msgs/Twist` | 선속도·각속도 명령 |
| `/odom` | Unity → ROS | `nav_msgs/Odometry` | 로봇 이동 추정 |
| `/tf` | Unity/ROS → ROS | `tf2_msgs/TFMessage` | 좌표계 관계 |
| `/scan` | Unity → ROS | `sensor_msgs/LaserScan` | 2D LiDAR |
| `/clock` | Unity → ROS | `rosgraph_msgs/Clock` | 시뮬레이션 시간 |
| `/camera/image_raw` | Unity → ROS | `sensor_msgs/Image` | 카메라 영상 |

## 4. 좌표계

ROS는 일반적으로 x 전방, y 좌측, z 상방의 오른손 좌표계를 사용한다. Unity는 x 우측, y 상방, z 전방의 왼손 좌표계를 사용한다. 위치·회전·쿼터니언·각속도를 그대로 복사하면 축 반전 문제가 생긴다.

첫 `/cmd_vel` 실습은 `linear.x`와 `angular.z`라는 스칼라만 사용하지만, `/odom`, `/tf`, 센서 데이터를 구현할 때는 명시적 좌표 변환 계층을 둔다.

## 5. 브리지 선택 경계

rosbridge는 JSON/WebSocket 변환을 사용하므로 교육, 상태 메시지, 제어 명령에는 적합하다. 다음 조건에서는 성능 측정 후 다른 전송을 검토한다.

- 고해상도 RGB·Depth 카메라 다수
- 대용량 PointCloud2
- 높은 주기의 다채널 LiDAR
- 강한 실시간성이나 DDS QoS 의미 보존이 필요한 제어
