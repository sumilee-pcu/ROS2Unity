# ADR-0002: 초기 저장소 공개 범위

- 상태: 권고
- 작성일: 2026-07-10

## 결정

교재 초고와 PoC 개발 기간에는 저장소를 Private으로 운영한다. 다음 조건을 충족한 시점에 공개용 저장소 또는 공개 브랜치를 분리한다.

## Private을 권장하는 이유

- POD 출간 전 원고의 무단 복제 위험을 낮춘다.
- 검증 전 코드가 완성된 공식 예제로 오인되는 것을 막는다.
- Unity 에셋, URDF, 이미지의 재배포 권리를 정리할 시간이 필요하다.
- 개인 환경 정보, IP, 녹화 화면, 수업 자료가 실수로 포함될 수 있다.

Private은 비밀정보 저장을 허용한다는 뜻이 아니다. 토큰, 키, 비밀번호는 저장소 공개 여부와 관계없이 커밋하지 않는다.

## 공개 전 체크리스트

- [ ] 교재 원고와 코드 라이선스 확정
- [ ] 외부 이미지·모델·URDF·에셋 출처와 재배포 권리 확인
- [ ] Git 기록에 토큰·IP·개인정보가 없는지 검사
- [ ] Unity 생성 폴더와 유료 에셋이 제외되었는지 확인
- [ ] 모든 실습의 검증 상태 표시
- [ ] 공개용 README, 인용 방식, 면책 문구 확정
- [ ] 출간 계약상 공개 가능한 범위 확인

## 공개 전략

권장 방식은 두 저장소 분리다.

1. Private `ROS2Unity`: 전체 원고, 실험 기록, 제작 자산
2. Public 예제 저장소: 검증된 코드, 축약 실습, 공개 가능한 그림

출판 정책이 코드 공개를 허용한다면 하나의 저장소에서 `docs/draft`만 제외하는 방식도 가능하지만, Git 기록에 초고가 남을 수 있으므로 저장소 분리가 더 안전하다.
